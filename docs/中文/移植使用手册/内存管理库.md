# 内存管理

有多种内存管理算法，如类SLOB链表、红黑树、位图内存池、链表内存池、区域内存管理

## 类 SLOB （heap）

一个轻量级分配器，支持动态内存申请、释放、统计与泄漏检测。

单链表管理空闲块，首次适配，按需切割，相邻块自动合并。

适用于嵌入式场景，代码小、碎片少、无锁。

### **API**

```
void heap_init(void);                         // 初始化堆
void *heap_malloc(size_t size);               // 分配内存
void  heap_free(void *ptr);                   // 释放内存
struct heap_stats heap_get_stats(void);       // 获取堆统计信息
void heap_debug_dump_leaks(void);             // 打印未释放的内存（调试）
```

若开启 `HEAP_TRACKING=1`，`heap_malloc` 和 `heap_free` 会记录文件名与行号，用于定位泄漏。

### **使用方法**

### **初始化**

```
heap_init();
```

在系统启动时调用一次即可。

### **分配与释放**

```
void *p = heap_malloc(100);   // 分配 100 字节
heap_free(p);                 // 释放
```

所有分配的内存必须由 `heap_free` 释放。

### **堆统计信息**

```
struct heap_stats st = heap_get_stats();
```

结构体内容：

```
remain_size     // 剩余可用空间
free_size_iter  // 遍历空闲链表得到的总空闲空间
max_free_block  // 最大连续空闲块
free_blocks     // 空闲块数量
```

用于监控内存碎片情况。

### 内存泄漏检测（可选）

当 `HEAP_TRACKING = 1` 时：

```
heap_debug_dump_leaks();
```

会打印所有未释放的内存块及其来源位置。

## **注意事项**

- 内存必须成对使用 `heap_malloc` / `heap_free`。
- 分配器基于空闲链表，适合嵌入式场景，不支持多线程并发访问（如需并发，请自行加锁）。
- 若申请失败返回 `NULL`，应检查堆是否耗尽或碎片过多。
- 堆大小由 `CONFIG_HEAP` 决定，可根据项目需求调整。



## 红黑树-链表版本

`memalloc` 是一个基于 **红黑树 + 双向链表** 的小型堆分配器，支持快速查找最大空闲块，适合需要更高分配效率的场景。

### **API**

```
void *mem_malloc(size_t size);   // 分配内存
void  mem_free(void *ptr);       // 释放内存
```

## **使用方法**

```
void *p = mem_malloc(128);
mem_free(p);
```

## **特性**

- **红黑树管理空闲块**：按 block_size 排序，可快速找到“最合适的空闲块”。
- **双向链表维护物理顺序**：释放时可检测相邻块并自动合并，减少碎片。
- **首次大于等于（best-fit）策略**：通过 `rb_first_greater()` 找到最小可用块。
- **按需切割**：分配后剩余空间会被切成新的空闲块并重新插入红黑树。
- **自动合并**：释放时向前、向后检查相邻空闲块并合并。
- **无锁**：适用于单核 MCU，如需并发访问请自行加锁。

## **注意**

- `mem_malloc` 返回的内存已按 8 字节对齐。
- `mem_free` 只能释放由 `mem_malloc` 分配的指针。
- 若申请失败返回 `NULL`，可能是堆耗尽或碎片过多。
- 不支持多线程并发访问（需要外部锁）。
- 默认堆大小可根据需求调整。

## **适用场景**

- 需要比简单链表堆（heap_malloc）更快的分配速度
- 需要更好的大块分配成功率
- 需要减少碎片的场景
- 适用于整体内存较大、分配频繁的场景
- 需要“best-fit”策略的系统组件

# 固定块内存池（membit）

`membit` 是一个**固定大小块（fixed‑size block）**的内存池，使用 32 位 bitmask 管理最多 32 个块。
 分配、释放均为 **O(1)**，适合频繁申请小对象的场景。

## **API**

```
membit_pool_t membit_create(uint16_t size, uint8_t count); // 创建内存池
void *membit_alloc(membit_pool_t pool);                    // 分配一个块
void membit_free(membit_pool_t pool, void *addr);          // 释放块
void membit_destroy(membit_pool_t pool);                   // 销毁内存池
void membit_reset(membit_pool_t pool);                     // 重置池（全部可用）
```

## **使用方法**

```
membit_pool_t pool = membit_create(64, 10); // 10 个 64 字节块

void *p = membit_alloc(pool);               // 分配一个块
membit_free(pool, p);                       // 释放块

membit_destroy(pool);                       // 销毁
```

## 特性

- **固定块大小**：所有块大小相同，适合对象池、消息结构体、节点缓存等场景。
- **O(1) 分配/释放**：通过 bitmask 查找最低可用位（`__builtin_ctz`）。
- **最多 32 个块**：bitmask 为 32 位。
- **按 8 字节对齐**：自动对齐到 8 字节边界。
- **无碎片**：固定块结构，不会产生堆碎片。
- **无锁**：适用于单核 MCU，如需并发请自行加锁。

## **注意**

- `size` 会自动按 8 字节对齐。
- `count` 最大为 32。
- `membit_alloc` 返回的指针必须由 `membit_free` 释放。
- 若池满，`membit_alloc` 返回 `NULL`。
- 内存池本身由 `heap_malloc` 分配，销毁时必须调用 `membit_destroy`。

## **适用场景**

- RPC 参数结构体缓存
- 文件系统节点缓存
- 协议栈 buffer 池
- 小对象频繁申请/释放的场景
- 需要 O(1) 分配性能的模块

# 固定对象池（mempool）

`mempool` 是一个**固定大小对象池**，一次性分配一整块内存，然后按固定步长切成多个节点，与前面使用位图设计不同，使用链表设计，对数量没有限制。
 适用于频繁申请/释放同一大小对象的场景，性能稳定、无碎片。

## **API**

```
pool_head_handle mem_pool_create(uint16_t size, uint8_t amount); // 创建对象池
void *mem_pool_alloc(pool_head_handle pool);                     // 分配一个对象
void mem_pool_free(pool_head_handle pool, void *ptr);            // 释放对象
void mem_pool_delete(pool_head_handle pool);                     // 销毁对象池
```

## **使用方法**

```
pool_head_handle pool = mem_pool_create(64, 20); // 20 个 64 字节对象

void *p = mem_pool_alloc(pool);                  // 分配
mem_pool_free(pool, p);                          // 释放

mem_pool_delete(pool);                           // 销毁
```

## 特性

- **固定大小对象池**：所有对象大小相同，适合节点、消息、结构体缓存。
- **无碎片**：对象池一次性分配，内部按固定步长切割。
- **O(1) 分配/释放**：通过链表管理空闲节点。
- **自动合并空闲链表**：释放时按 next 指针寻找合适位置插入。
- **按 8 字节对齐**：自动对齐到 8 字节边界。
- **无锁**：适用于单核 MCU，如需并发请自行加锁。

## **注意**

- `size` 为用户对象大小，不含内部头部；实际分配会自动对齐。
- `amount` 为对象数量，池大小固定，不会动态扩容。
- `mem_pool_alloc` 返回的指针必须由 `mem_pool_free` 释放。
- 若池满，`mem_pool_alloc` 返回 `NULL`。
- 对象池本身由 `heap_malloc` 分配，销毁时必须调用 `mem_pool_delete`。

## **适用场景**

- 文件系统节点缓存
- RPC 参数/结果结构体池
- 协议栈 buffer 池
- 任务控制块（TCB）或小型对象的快速分配
- 需要稳定 O(1) 分配性能的模块

# 区域分配器（mg_region）

`mg_region` 是编译器ccBPF与运行时使用的**区域分配器（Region Allocator）**，支持两种模式：

- **Pool 模式（MG_REGION_POOL）**：按 2^N 大小分类的小块分配器，适合大量小对象（AST、IR、类型系统等）,基于membit内存池设计。
- **Bump 模式（MG_REGION_BUMP）**：线性分配，不支持 free，适合一次性构建、一次性释放的场景（如编译阶段）。

## **API**

```
mg_region_handle mg_region_create_pool(uint8_t max_class_bits); // 创建池化区域
mg_region_handle mg_region_create_bump(size_t cap);             // 创建 bump 区域
void *mg_region_alloc(mg_region_handle r, size_t size);         // 分配内存
void  mg_region_reset(mg_region_handle r);                      // 重置区域
void  mg_region_destroy(mg_region_handle r);                    // 销毁区域
void  mg_region_print_pools(mg_region_handle r);                // 打印统计信息
```

## **使用方法**

### **Pool 模式（适合大量小对象）**

```
mg_region_handle r = mg_region_create_pool(12); // 最大类 2^12 = 4096 字节

void *p = mg_region_alloc(r, 128);             // 自动选择合适的 class
mg_region_reset(r);                            // 所有对象一次性释放
mg_region_destroy(r);
```

### **Bump 模式（适合编译阶段）**

```
mg_region_handle r = mg_region_create_bump(4096);

void *p = mg_region_alloc(r, 64);  // 线性分配
mg_region_reset(r);                // off = 0，全部可重用
mg_region_destroy(r);
```

## **特性**

### **Pool 模式**

- 按 **2^N** 分类（class 0~max_bits）
- 每个 class 内维护多个 **membit 池**
- 小对象自动进入最合适的 class（best-fit）
- 每个池的块数量根据大小自动选择（8/2/1）
- 分配/释放均为 **O(1)**（来自 membit）
- 适合编译器的 AST、IR、类型节点等生命周期一致的对象

### **Bump 模式**

- 线性增长指针（off）
- 不支持单独 free
- `reset()` 即可瞬间清空
- 适合“构建一次 → 全部丢弃”的阶段性内存

## **注意**

- Pool 模式的最小分配单位这些可以自己通过宏调整。
- Bump 模式不支持 free，必须通过 `reset()` 或 `destroy()` 回收。
- mg_region 本身由 `heap_malloc` 分配，销毁时必须调用 `mg_region_destroy`。
- Pool 模式内部使用 membit，不会产生碎片。

## **使用的场景**

- 短周期内存区域适用于大量小对象频繁使用的场景，本质上是内存池与复用
- 长周期内存区域适用于任何“分配多、释放少”的场景
