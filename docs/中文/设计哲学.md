## 引言

lttit是一个可动态编程的分布式操作系统。

## 构成

先简单介绍GitHub上的工程目录，方便感兴趣的读者阅读源码或者移植组件。

综合来看，lttit其实可以拆分为十几个组件：

GitHub上的主目录，一个demo是0.1版本的双节点运行，lttit则是工程目录：

构成如下：

```c
├─ccBPF  
│  ├─compiler   编译器
│  │  ├─backend  后端
│  │  ├─frontend 前端
│  │  └─ir       中间形式
│  └─vm          
│      └─bpf     bpf虚拟机
├─cluster  全局节点树
├─CSC      分布式通信协议栈
│  ├─ccnet  路由协议
│  ├─ccrpc  远程调用协议
│  └─scp    可靠协议
├─fs      文件系统
├─lib     数据结构和数学算法库
├─mg      内存管理
```

## 设计哲学

### 空间与时间的抽象

本地和远程，其实是人为划分的概念，我们通过网络或者其他方法发送消息给远程，与我们直接通过硬件总线通信，其实只有空间与时间的区别：

1.本地近，远程远

2.本地快，远程慢

但是，如果远程可以做到跟本地差不多快呢？

那么，我们完全可以把内存都挂在远程，直接从远程读取指令执行。

这甚至意味着我们可以拿多计算机当成多核计算机来用。

所以，本地和非本地只是人为划分的概念，只有空间与时间的量的差距，而无质的差别。

#### 传输

对于数据传输场景，uart、spi、iic、网卡等等，在我们的抽象下，这些传输也没有区别。

我认为数据传输也是空间与时间的转移：

1.数据A从地点P1到达了P2

2.传输花了多久

所以，我们的通信协议只需要做到两件事：

1.把数据完好无损送到目的地

2.尽量快

### 激进的分布式

lttit的分布式其实非常激进，大部分操作系统的分布式都是网络通信的扩展，很少会有操作系统打算直接让组件也分布式化，这样意味着不确定性。

但是，lttit的设计就是一种对空间和时间的抽象，本地和远程其实没什么区别。

由于各个组件是模块化的，我们压根不需要在同一台设备上运行整个lttit，其实我们可以这样干：

1.nodeA运行文件系统

2.nodeB运行TCP/IP，驱动网卡

当nodeA想上网时，通过分布式总线发消息给nodeB，nodeB看到后，会把消息发到公网上，等消息回来后，根据记录，把消息发给nodeA。

当nodeB需要持久化记录某些数据时，通过分布式协议总线直接把数据写入到nodeA的文件系统里面。

```
        +------------------+
        |     Leader       |
        |  (World Tree)    |
        +--------+---------+
                 |
     --------------------------------
     |                              |
+----+-----+                  +------+-----+
|  Node A  |                  |  Node B    |
|  FS      |                  |  TCP/IP    |
|  BPF VM  |                  |  Drivers   |
+----------+                  +------------+
```



由于每一个节点的驱动和外设也不同，nodeA和nodeB都把自己的设备注册到它们中的一个节点，这个节点被称之为leader。

这种激进的分布式在通用操作系统领域是灾难，但是对于嵌入式这些稳定的内网环境来说，可能还会有一点用武之地。

### 动态改变

对于每一个节点来说，可能需要调整部分逻辑，在传统开发中，这往往需要重新编译并且烧录，要么就是定义配置，然后切换。

其实不用这么麻烦，lttit有bpf编译器和虚拟机，实现了类似Linux内核的eBPF的功能，我们可以在任意可能会更改逻辑和配置的地方加上hook点，这些hook点可以注册任意c函数，同时对程序进行验证以确保不会干掉内核。

当我们需要修改某些配置时，我们只需要重新编程，然后注入可执行文件到目标节点的hook点即可。

我们的编译器和虚拟机也是分离的，这意味着可以在nodeA上编译，然后发送可执行文件到nodeB上，然后nodeB注册对应的hook程序。

比如，我们需要动态设置防火墙或者统计流量、甚至是限制流量，那么，我们只需要修改一下hook程序就行了。

### 一切皆文件

对于使用者来说，不同节点就是一个文件目录，所有本地和远程资源以文件形式组织。

比如/nodeA、/nodeB，各种设备都被抽象为了基本的文件，比如：

/nodeA/led，由各个节点自行注册对应的led操作。

这个抽象与plan9一模一样，很早我有这个设计idea，后面我很惊讶plan9最开始就这么干了。

（plan9被称之为世界上最好的操作系统，但也是一个注定会失败的操作系统）

### 微内核

lttit必然是需要微内核设计的，这个不用过多解释了。

微内核并不意味着简单，有时候，越空白才能留下更多。

### 一切交互都是消息传递

我们的各个组件、各个模块，其实都是一种消息传递的交互模型。

A节点甚至是A模块输入了什么，必然会输出些什么。

整个世界也遵从这个模型，所以：程序是模块交互的总和，而消息总线则是背后交织的网。



## 设计哲学的实现

## 分布式

得益于微内核与分布式通信协议设计，分布式操作系统需要满足以下特点：

- 单一系统映像： 整个集群看起来像一台机器：统一的文件系统、统一的资源视图、统一的接口。
- 透明性：用户不需要关心资源位于哪个节点，包括：
  - 位置透明
  - 访问透明
  - 迁移透明
  - 扩展透明
- 资源共享：多节点共享 CPU、内存、设备、文件、网络等资源。
- 通信与协调：  节点之间通过消息传递、RPC、事件等方式协作。
- 独立节点：每个节点都是独立的计算单元，有自己的 CPU、内存、驱动。

在我们的cluster中，我们所有的资源其实都挂在了一颗前缀树上，这给使用者提供了一个单一的视图。

```
├─cluster  全局节点树
```

### leader与member

在所有节点中，会设置一个leader，它会保存全局世界树，当member加入lttit时，会发送写世界树的申请，把自己要提供的资源以文件的形式挂在世界树上。

所有的资源都遵从文件接口：

```c

struct vfs_ops {
    int (*open)(void *self, const char *path, int flags);
    int (*read)(void *self, int fd, void *buf, int len);
    int (*write)(void *self, int fd, const void *buf, int len);
    int (*ctl)(void *self, int fd, int cmd, void *arg);
    int (*close)(void *self, int fd);
};
```



## 动态与安全性

lttit中的编程，是通过编译器和一个bpf虚拟机实现的。

编译器编译c程序，得到可执行文件，然后虚拟机读取并执行。

对于运行中内核的编程组件来说，安全性反而是第一位的。

所以，ccbpf的设计追求的是可验证。

我们的编译器其实就是一个c子集，而虚拟机则需要限制指令。

### 微内核设计

我们的内核是一个RTOS，但是，整个lttit其实有点软实时。

由于OS是由微内核调度实现的，因此用户编写的进程既可以作为应用程序，也可以作为底层OS的进程。

微内核有内核调用以支持以下内容：

* 线程
* 时钟
* 定时器

* 内存管理
* 中断处理
* 信号量 、互斥锁等ipc

## 传输

在lttit中，担任通信总线以及远程调用功能的是协议栈CSC，它有三个协议，干了三件事情：

```
├─CSC      分布式通信协议栈
│  ├─ccnet  路由协议
│  ├─ccrpc  远程调用协议
│  └─scp    可靠协议
```

1. ccnet，一个简单的dijkstra算法，把数据包送到目的地并且检验完整性
2. scp，一个有点复杂但性能不错的可靠协议，负责实现拥塞控制、流量控制，尽量在一定时间内送更多的数据给对方，而且保证不打爆对方

3. ccrpc，一个远程调用协议，实现了基于文件接口的远程调用，是一切皆文件的体现，这意味着任何操作都要写成文件相关的接口，也就是：打开、读取、写入等。
